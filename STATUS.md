# Continuum — Status

## What it does

Continuum is a Chrome extension that captures an AI chat session and produces a handoff (Markdown / PDF) that can be resumed in a fresh chat without context loss. **Capture** runs on **Claude (claude.ai)**, **ChatGPT (chatgpt.com)**, and **Gemini (gemini.google.com)**; Perplexity is planned. Gemini capture is **DOM-based** (no clean API — see the Gemini adapter note) and its selectors are probe-tuned. As a **resume target** (where a captured chat is re-opened), Claude & ChatGPT are wired; Gemini & Perplexity resume-picker rows stay placeholders — but a Gemini-*sourced* session can already be resumed INTO Claude/ChatGPT (source ≠ target).

### Multi-AI architecture (adapter routing)
The provider-specific bit is the **capture adapter**, chosen by hostname via `Continuum.getActiveAdapter()` (`content-script.js`): `claude.ai → claudeAdapter`, `chatgpt.com`/`chat.openai.com → chatgptAdapter`, `gemini.google.com → geminiAdapter`. **Gemini adapter (`src/adapters/gemini-adapter.js`)** is DOM-based: Gemini has no replayable REST endpoint (obfuscated `batchexecute` RPC), so `capture` scrapes the rendered conversation — `collectTurnEls` walks `div.conversation-container` → `<user-query>` / `<model-response>` (selectors in `SEL`, comma-lists with fallbacks, tuned via `geminiAdapter.probe()`), `scrapeAttachments` pulls content `<img>`s (size/avatar-filtered) and file chips. Image fetching goes **via the background worker** (`continuum-fetch` in `background.js`) since Gemini media lives on `lh3.googleusercontent.com` (a content-script fetch is CORS-blocked). **Caveat (probe-confirmed 2026-06):** Gemini's uploaded-image previews + generated images use **token-gated** `/gg/…→/rd-gg/…` URLs that **403 every separate fetch** — with/without cookies, and even with a spoofed `Referer` (a `declarativeNetRequest` rule was tried and removed; it didn't help). The `<img>` renders them via a one-time page-scoped grant we can't replay, so those images are captured as **name-only references** (`[image: … — not captured]`). Source/citation thumbnails (`sources-carousel-inline`, `gstatic.com`) are filtered out so they don't inflate the image count. Image **counting** is unaffected (images count even when name-only); **files** follow the per-provider rule — Gemini counts files YOU uploaded plus any with a real download link, while AI-generated link-less file cards stay name-only (Claude stays strict: attachable-only). Generated **file cards** ("Your PDF file is ready" → filename + type badge + Open button) match no stable selector, so `findFileCards` locates them via their Open/Download control and `pickFileName` rebuilds the name from the visible name + type badge (e.g. `…images` + `JSON` → `images.json`); they're name-only refs unless a real download link exists. **Two load quirks handled:** (1) Gemini renders the app shell first and fetches turns a beat later, so `waitForConversation` waits until the turn count is non-zero and STABLE before reading (NOT for the `chat-loading-animation`, which stays in the DOM permanently); (2) Gemini **virtualizes** history (only on-screen turns mount), so `loadAllTurns` scrolls the `infinite-scroller` to the top until the count stops growing — capturing the WHOLE chat, not just the visible part. `captureFast` defers to `capture`; `isConversationPage` treats Gemini `/app/<id>` as a conversation. The panel (`capture-panel.js` `activeAdapter()`) and content-script probes route through it, so the UI is provider-agnostic. Each adapter exposes the same surface (`captureFast`, `capture` DOM fallback, `detectTitle`, `detectStartedAt`, `peekStats(Fast)`, probes). `content-script.isConversationPage()` is site-aware (claude `/chat/<id>`, chatgpt `/c/<id>`). Sessions carry `sourceProvider` (`claude`/`chatgpt`) → folder grouping + logo. **Two id axes, kept distinct:** AI identity (`claude`/`chatgpt`, for capture/folders/resume-target) vs compression vendor (`anthropic`/`openai`/…). The **ChatGPT adapter** (`src/adapters/chatgpt-adapter.js`) gets a bearer token from `/api/auth/session`, fetches `/backend-api/conversation/<id>`, walks the message `mapping` from `current_node` → root for the active path, extracts text + attachments (uploads via `metadata.attachments`, inline images via `image_asset_pointer`; bytes via `/backend-api/files/<id>/download`), with a `[data-message-author-role]` DOM fallback. **Unlike Claude, ChatGPT serves ALL upload types back** (zip, json, any binary) via the files download endpoint, so the ChatGPT adapter keeps every `metadata.attachments` entry (no `.zip` skip) and `captureFast` fetches its bytes by `file_id` → they're captured + attachable. Exact fields are probe-verified (`probeMessages`/`probeApiAttachments` on the active adapter).

## Capture: Fast is the default; DOM is the fallback

**Fast capture (`captureFast`, API path) is the only user-facing path** — capture mode is no longer a setting (the panel toggle was removed). The panel's "Capture this session" always calls `captureFast`; DOM capture (`capture`) survives solely as the automatic fallback `captureFast` invokes when the API call fails.

- **Fast capture (`captureFast`)** — pulls the conversation from claude.ai's REST endpoint (`?tree=True`).
  - `resolveActivePath` walks `parent_message_uuid` from the current leaf to keep only the live conversation, dropping **off-path branches** from edits/regenerations (this is why a raw tree showed 413 messages where the real chat had ~367). Empty nodes are filtered. Unit-tested in `tests/adapter-tree.test.js`.
  - Attachments are the user's **uploads only** — read straight from each API message's `files`/`attachments`/`files_v2` arrays (`apiMessageAttachments` → `classifyApiAttachment`), so Fast reliably captures uploaded files the DOM path misses. (Capturing media **Claude itself generates** — the "Download" document cards, generated images — was built and then **removed** by request; those assets live in the assistant content blocks, and in `rendering_mode=raw` claude.ai collapses them to a placeholder with no fetchable URL anyway. If it's ever wanted again, it would re-read the content blocks.) Blobs are fetched **in parallel** (`mapLimit`, 6 concurrent). **Text-file uploads have their contents inlined** into the transcript; binary/image bytes are stored for the ZIP. **Byte-URL resolution is generic** (`classifyApiAttachment`): it tries the known shapes first (top-level `*_url`, then nested `document_asset`/`preview_asset`/`thumbnail_asset`/`file_asset``.url`), then falls back to **any** property ending in `_url` or any asset-ish object carrying a `.url` — so arbitrary file kinds (zip, JSON exports, other binaries) resolve their bytes instead of only PDFs/images. **`.zip` uploads are no longer hard-skipped** (the old "unservable code-sandbox blob" exclusion is gone): a zip is fetched when a URL resolves, else it degrades to a name-only reference (rendered `[file: name — not captured (bytes unavailable)]`). **Confirmed via probes (2026-06): `file_kind:"blob"` uploads (zip, JSON exports — anything sent to the code-interpreter tool, path `/mnt/user-data/uploads/…`) carry NO download URL — only a `file_uuid` — and are NOT retrievable.** The canonical contents route `/api/{org}/files/{uuid}/contents` returns a user-facing `"File not found"` for these uuids (11 endpoint variants tried, all 404), so the bytes live only in the ephemeral sandbox and can't be re-downloaded by anyone via the web API. Regular uploads (images via `preview_url`/`preview_asset`, PDFs/docs via `document_asset.url`, text via `extracted_content`) capture fine. **The file count + "Attach files" toggle count ONLY attachable files** — `recomputeStats` counts a file only when its bytes were captured (`mediaId` set), and both adapters' `peekStatsFast` previews mirror that (Claude: file needs a `url` or inline text; ChatGPT: needs a `file_id`/sandbox path). So a Claude un-fetchable blob is referenced in the transcript but contributes 0 to the count/toggle, while every captured ChatGPT upload counts. Diagnostics: `probeApiAttachments()` dumps attachment shapes; `probeFileDownload()` (flag `continuum_probe_dl`) tries candidate byte-URLs for the first URL-less file and reports status/body.
  - **Raw-mode placeholder is stripped.** In `rendering_mode=raw`, claude.ai replaces any media block it can't render as text (generated images/visuals/artifacts) with the literal *"This block is not supported on your current device yet."* `cleanAssistantText` removes that placeholder (and an empty ```` ``` ```` fence wrapping just it) **line-by-line** — line-based so a real adjacent code block is never eaten — so it doesn't pollute the transcript. (We don't capture that media; the string is pure noise.)
  - **All content-block types are represented in the transcript, never silently dropped** (`extractMessageText`). It processes the structured `content` blocks and represents each: `text` → the answer; tool use (web search / code / other) → `_[Web search]_ <query>` / `_[Code: …]_` / `_[Tool: …]_`; tool/search results → `_[Search results (N)]_` + result titles+links; any unrecognized type → `_[<type>]_`. **Thinking blocks are skipped entirely** (not labeled, not included) — a handoff needs Claude's final answers/code/decisions, not its internal reasoning; thinking is long (bloats the resume PDF) and its sub-shapes (thinking/summary/redacted) are fragile, so dropping the whole class is cleaner than chasing fields. An assistant turn that is *only* thinking becomes empty and is filtered as a placeholder node. Non-answer parts are labeled prose, kept verbatim in the recent tail and trimmed by `extractKeyContent` on older turns. Field names follow the public Anthropic block shapes; if claude.ai's internal shape differs, a block falls back to its bare `_[<type>]_` label — run **`Continuum.claudeAdapter.probeMessages()`** in the console (now recurses into blocks + dumps file arrays, prefers turns carrying media) to confirm/tune the exact fields. Only the API path sees these blocks; the DOM fallback doesn't.
- **DOM capture (`capture`)** — scrolls to mount lazily-loaded turns, then scrapes rendered markdown. `ensureFullRender` sweeps in overlapping 60%-viewport steps with per-stop stabilization and detects destructive virtualization (tagged `…-virtualized`, with a panel warning). Kept thorough on purpose, so it's slower.

Known DOM gap (not pursued, since Fast is the path): `collectFiles` finds file uploads via `data-testid`-is-filename, which can read **0 files** when claude.ai changes the file-card markup or the cards aren't mounted. Fast capture is unaffected (reads files from the API). The assistant-turn selector (`[data-is-streaming]`) is the other recurring fragility point — see the `continuum-assistant-selector-fragility` memory.

## Compression — LLM "summarize the middle" (opt-in, resume only)

The old rule-based compressor was deleted; compression is now **LLM-based** and runs only on the **resume** path (the `conversation-history.pdf` + clipboard text). The ZIP and "Copy chat history" stay fully verbatim.

**Model (`src/core/llm-compressor.js`):** `compressSession(session, {keepCount, apiKey, model, onProgress})` keeps the **first N and last N messages verbatim** and replaces the **middle** with a single LLM summary. `sliceTurns(turns, keepCount)` (pure, unit-tested) splits top/middle/bottom; the middle is rendered to transcript text via `buildHandoff` and sent to the Anthropic API; `assembleCompressed` returns a cloned session whose `turns = [...top, summaryTurn, ...bottom]`. The `summaryTurn` has `role:"summary"` + `omittedCount`; `buildHandoff` renders it as a `## Summary of N condensed messages` block. Short chats (`turns ≤ keepCount*2`) return unchanged. **Middle IMAGES and FILES are preserved as references:** the LLM only condenses text, so `collectMiddleAttachments` (exposed as `collectMiddleImages` / `collectMiddleFiles`) carries every middle image AND file attachment (deduped by `mediaId`, skipping any already in the verbatim top/bottom) onto `summaryTurn.attachments`. `buildHandoff`'s summary branch then appends two blocks after the summary text: `### Images from condensed messages` (each `_[Image: filename]_` caption + `![](…)` ref — so `buildResumePdf` still embeds the picture, which it couldn't once the middle turns were replaced) and `### Files from condensed messages` (each `[file: name → files/name]` reference — listed, not re-inlined, since the file's contents were already part of what the LLM summarized). In **PDF** mode the carried file bytes also ride along as attachments (if "Attach files" is on, via `collectResumeDocuments`); in **Markdown** mode everything is reference-only. The summary header renders as `## Compressed N messages`. To keep the summary prose clean, `compressSession` runs `stripImageRefs` on the middle text before summarizing — inline `![alt](…)` refs become plain `[image: alt]` hints so the LLM can't emit a dead/colliding markdown ref.

**Multi-provider via a background service worker.** The summarization call is made by **`src/background.js`** (MV3 service worker), not the content script — a worker fetch uses the extension's `host_permissions` and **bypasses CORS**, which is required for providers that block direct browser calls (OpenAI, Perplexity). `llm-compressor.summarizeMiddle` `chrome.runtime.sendMessage`s `{type:"continuum-summarize", provider, apiKey, model, system, text}`; the worker hits the right endpoint and returns `{ok,text}` or `{ok:false,error}`. Supported providers + default models: **anthropic** (`/v1/messages`, `claude-haiku-4-5-20251001`), **openai** (`/v1/chat/completions`, `gpt-4o-mini`), **gemini** (`generativelanguage…/:generateContent`, `gemini-2.0-flash`), **perplexity** (OpenAI-shaped, `sonar`). System prompt preserves code/files/URLs/numbers/decisions/state, drops filler, never invents. `manifest.json` registers the worker and lists all four API hosts in `host_permissions`.

**Settings (`settings.js`):** `compressProvider` (anthropic|openai|gemini|perplexity, default anthropic), `compressApiKeys` (a per-provider key map stored in `chrome.storage.local`; the legacy single `anthropicApiKey` is migrated into it), `compressKeepCount` (default **8**, clamped **5–10**). Settings view: a **provider dropdown** + an **API-key field** (shows/saves the selected provider's key) + keep-count. Default model is chosen per provider by the worker (no model picker yet).

**Trigger / fallback:** a **"Compress with AI"** toggle in the session detail view → `onResume` marker `compress:true`. `resume-injector.checkPendingResume` reads the active provider + its key, swaps in the compressed session before `buildResumePdf`. **Always falls back to the full verbatim session** — no key → hint toast; too-short chat → note; API/network/worker error → the **actual error is shown in the toast** + `console.error` (run in the *new* tab) — compression never blocks the resume. The success toast appends `· compressed N→M tokens (−X%)`. The token figure is **per-format** (`payloadTokens` in `resume-injector`): `estimateTokens(text)` PLUS, for **PDF only**, `~1400 × embedded-image-count` (the model's vision cost to *see* each embedded image). **Markdown adds nothing for images** (it only references them), so the same chat reads as fewer tokens in MD than PDF — which is the real difference between the formats. Target ~30–55% is a goal, not a guarantee (verbatim ends bound it).

**Tested:** `tests/llm-compressor.test.js` covers `sliceTurns`/`assembleCompressed` (the network call isn't exercised in unit tests).

### Token estimate (kept)
There is no exact offline tokenizer for current Claude models, so `estimateTokens` is **`chars / 4.35`** — calibrated against `gpt-tokenizer` (a close BPE proxy), ~6% mean abs error across the corpus (beats flat `chars/4` ~7.7%). Used for the before/after % readout and `eval-gate.js`.

It does not yet drive the LLM compressor (that needs a live API key); the totals are the "before" baseline.

## Resume → auto-fill the new chat

"Resume in new chat" no longer just copies to clipboard — it **auto-fills the freshly-opened chat** so the model has full prior context:

- **AI picker (resume target):** "Resume in new chat" expands an **inline list** of target AIs. **Claude, ChatGPT, and Gemini are wired up** (`onResume(target)` opens that AI's new-chat URL via `RESUME_URLS` — `claude → claude.ai/new`, `chatgpt → chatgpt.com/`, `gemini → gemini.google.com/app`); the resume-injector then fills that site's composer (`COMPOSER_SELECTORS` cover ProseMirror, `#prompt-textarea`, and Gemini's Quill `rich-textarea .ql-editor`). Perplexity/Grok/etc. stay disabled placeholders ("soon"). The resume *target* is independent of the *compression provider*. **Gemini file caveat:** Gemini's upload is gated behind an OS file picker (its "+" opens a menu; no settable hidden `input[type=file]`) and ignores synthetic drag-drop — so on Gemini the injector **DOWNLOADS** the handoff file(s) to the user's Downloads (`downloadFiles` — object-URL + `<a download>`) and the toast tells them to attach via "+→Upload files"; auto-send is skipped there. Claude/ChatGPT still attach programmatically.
- **Format selector (PDF vs Markdown):** the detail view has a **PDF / MD radio pair** (`.cn-radio` pills in `.cn-format-select`, default **PDF**) below the Resume button, with a short pros/cons hint ("PDF embeds images and references files — heavier, model can see images; MD references images/files by name only — lighter, fewer tokens, text-only"). You pick one before clicking Resume; the choice rides as `marker.format` (`"pdf"`|`"markdown"`). Format is **independent of compression** — compression runs first in the injector, then the format decides packaging, so either format can be AI-compressed. **PDF (default):** `conversation-history.pdf` with images embedded inline (below). **Markdown:** `resume-injector` skips the PDF build and ships `conversation-history.md` (the verbatim/compressed handoff markdown), which **references** images as `![](images/…)` and files as `[file: … → files/…]` by name. **Two attach toggles (per-format, via `updateAttachRows`):** an **"Attach N files"** row (documents, both formats — shown when the chat has files) and an **"Attach M images"** row (**Markdown only**, since PDF already embeds images — shown when the chat has images). Both default **OFF**, ride as `marker.includeFiles` / `marker.includeImages`. For **PDF** the files toggle gates the documents that ride alongside the embedded-image PDF. For **Markdown**: toggles both OFF = just the `.md` (lightest, fewest tokens); each ON independently attaches that kind's referenced bytes. **No cap** — everything the user opts into is attached (claude.ai may reject a batch over its own ~20-attachment limit, but that's the user's call). The resume message is **format-specific** (see the Preamble bullet): Markdown uses the separate `resumePreambleMd` setting (default `DEFAULT_RESUME_PREAMBLE_MD`). The **PDF-build failure fallback** shares the Markdown attach branch (`!usedPdf`): it ships the `.md` PLUS images (ALWAYS — the PDF would have embedded them) + documents (gated by the files toggle).
- **Hand-off:** `onResume(target)` (`capture-panel.js`) writes a marker `{ sessionId, includeFiles, compress, format, target, ts }` to `chrome.storage.local` (`continuum.pendingResume`) and opens the target's URL. `resume-injector.checkPendingResume` runs on whichever site the new tab is (claude.ai or chatgpt.com) and fills that site's composer — `COMPOSER_SELECTORS` cover ProseMirror (Claude) + `#prompt-textarea` (ChatGPT), `SEND_SELECTORS` cover both, file-attach is generic (`input[type=file]` + drop). The clipboard copy stays as a fallback.
- **Injection:** `src/resume-injector.js` (`Continuum.resumeInjector.checkPendingResume`, called from `content-script.js` `init()`) reads+clears the marker (TTL 2 min), loads the session from IndexedDB, **fills the composer with the editable preamble**, and **attaches `conversation-history.pdf` plus any binary documents as separate files**. It does NOT submit — the user reviews and sends.
- **Resume PDF (`src/core/pdf-export.js`, vendored `jspdf`):** claude.ai can't read inside a `.zip` in-chat, and a plain markdown can't carry images as vision. The PDF carries the transcript text **and** the images: `buildResumePdf` lays out the handoff markdown as monospaced text (Courier keeps diagrams/code aligned), paginated, and **embeds each image inline** at its `![…](images/…)` reference. The text has cosmetic Markdown stripped for clean prose via the shared **`cleanHandoffMarkdown`** (re-levels `## User`/`## Assistant`, drops other `#` heading marks, removes `**bold**`/`_italic_`/`` `code` `` backticks, peels the `_Captured…_` line markers; keeps fenced ``` blocks). **The resume `.md` applies the SAME `cleanHandoffMarkdown`** (in `resume-injector`), so the PDF and the `.md` read identically — that was the requirement. (`buildHandoff` keeps the raw Markdown, so the ZIP `transcript.md` + eval-gate parser are unaffected.) Images are still pulled out upstream and embedded **in transcript order/flow** (decoded via canvas → JPEG, since captures are often webp and jsPDF won't take webp). Text-file contents are already inlined in the markdown. The PDF is built with jsPDF `compress: true` (deflate-compressed streams — smaller upload; this is file compression, unrelated to any transcript/token compression). **Scale handling (image-heavy chats):** images are downscaled to ~1400px and JPEG-encoded (q0.8) before embedding — full-res PNGs would make a 50–100 MB PDF claude.ai rejects; downscaling brings it to a few MB and Claude's vision downsamples to ~1568px anyway, so nothing visible is lost. The build yields to the event loop periodically (no tab freeze), shows a progress toast, finds the composer *before* building (fail fast), and logs `[Continuum] resume PDF: <MB>, <images embedded>`. If the build fails, the fallback caps separate image attachments at 20.

**Documents (PDF/DOCX/…) — attached SEPARATELY, never merged.** Per directive, only **images** are embedded into `conversation-history.pdf`; binary documents ride alongside as their **own attachments** (claude.ai reads multiple files fine), and the transcript references them by name. `Continuum.pdfExport.collectResumeDocuments(session)` gathers **every byte-backed file attachment** (PDF, DOCX, XLSX, and also text files whose contents are inlined — the user may still want the real file; images excluded), de-duped by `mediaId`, in transcript order; the resume injector concatenates them after `conversation-history.pdf`. (It includes text files so the count matches the panel's file stat and the "Attach files" toggle.) A document's bytes come from `classifyApiAttachment` resolving its URL (top-level `*_url` or nested `document_asset`/`preview_asset`/`thumbnail_asset.url`, with stray newlines stripped from filenames). The old **pdf-lib page-merge is removed** and `pdf-lib.min.js` is no longer loaded (dropped from the manifest). **"Attach files" toggle:** the detail view shows a checkbox below the Resume button — **only when the chat has files** (gated on `session.stats.files > 0`, matching the visible file stat); it defaults **off** (opt-in), its label shows the count ("Attach N files to the new chat"), and its state rides to the new tab as `marker.includeFiles`. When off, the resume attaches the PDF alone (images still embedded); when on/absent, the documents are concatenated after it. If PDF generation fails entirely, it **falls back** to attaching `conversation-history.md` + images + files separately (`Continuum.handoff.collectResumeFiles`).
- **Auto-send (opt-in):** `autoSendOnResume` setting (default **off**), toggled via Settings → "Auto-send after resume". When on, after filling + attaching, `clickSendWhenReady` waits for the Send button to become **enabled** (claude.ai disables it while the attachment uploads = the safe "upload finished" signal), then clicks it — with a 2s initial delay and a 90s timeout (won't fire if it can't confirm, so it never sends without the file). Send-button selectors live in `SEND_SELECTORS`; `probeComposer()` now also dumps them.
- **Preamble setting (per-format):** two messages — `resumePreamble` (PDF, default `DEFAULT_RESUME_PREAMBLE`, names `conversation-history.pdf` + "images embedded inline") and `resumePreambleMd` (Markdown, default `DEFAULT_RESUME_PREAMBLE_MD`, names `conversation-history.md` + "images/files referenced by name, not attached"). Settings → "Resume message" has a **PDF / Markdown radio toggle** (`.cn-radio` pills) that swaps which one the textarea edits; the textarea's value is held per-format in `_preambleDrafts` (synced on `input`) so flipping the toggle is instant and preserves unsaved edits, and **Reset to default** + the save toast act on the selected format. The injector picks `resumePreambleMd` vs `resumePreamble` by `marker.format`.
- **"Copy chat history" = clean shareable format.** `onCopy` uses `buildReadableTranscript` (not `buildHandoff`): a `# title` + `_Claude · N messages · date_` header, then **bold `**You**` / `**Claude**` labels** above each message, blank lines between turns, and tidy attachment refs (text-file contents inlined in a fenced block; images/binaries as `_[Image: …]_` / `_[File: …]_`). Deliberately separate from `buildHandoff` (the machine `## User/## Assistant` format the Resume PDF, ZIP, and eval-gate parse). Both are verbatim today.
- **Fragility:** composer text-injection (ProseMirror `execCommand insertText`, paste fallback) and file-attach (hidden `input[type=file]` via DataTransfer, drop fallback) depend on claude.ai's live DOM — same breakage class as the capture selectors. `Continuum.resumeInjector.probeComposer()` dumps current selectors (or set `localStorage.continuum_probe_composer = "1"` and reload); the clipboard fallback means a failed inject never strands the user. No new permissions (reuses `storage` + IndexedDB); `jspdf.umd.min.js` (~412 KB) is bundled as a content script.

## Known limitations

- **No compression yet.** The handoff is fully verbatim, so a very long chat produces a large resume payload (many PDF pages). This is the temporary state until the LLM compressor is built — it's the whole reason for the redo.
- **English-only patterns** *(applies to the upcoming compressor's heuristics, if any)*.
